## Pivot — module journal de trading (juin 2026)

> Bascule de l'app : d'un outil d'analyse de marché par ticker (narratif LLM) vers un **journal de trading**. L'unité atomique devient le **trade entry**.

### ✅ Livré — Page Candidats (cockpit de préparation d'un short : sizing au risque + exécution + cover) (2026-06-19, révisé 2026-06-21)

- **Nouveau module backend `candidates/`** (hexagonal) — `Candidate` : un candidat = un ticker préparé pour une séance (paramètres capital/risque, open, stop %, données du jour float/volume/push/borrow, **legs `fills` / `entries` / `exits` en JSONB**). **Cycle de vie piloté par la date** (`trading_date`) : pas de colonne statut — la dropdown ne liste que les candidats du jour, les antérieurs restent en base mais hors picker. `CandidateService` : **upsert keyé sur `(date, ticker)`** (changer le ticker vise un autre candidat, pas d'écrasement), `listForDate` (défaut aujourd'hui), validation in-service (ticker non vide, open / capital > 0, % risque ∈ (0,100]) → 400 ; contrat 404-pas-403. Les ladders ridant en JSON sont (dé)sérialisées via `ObjectMapper` + `TypeReference` (Hibernate hors du JSON, convention `screener_snapshot_day` / `ticker_narrative_snapshot`). `CandidateController` : `GET /api/candidates?date=`, `GET /{id}`, `POST`, `PUT /{id}`, `DELETE /{id}`. Migration **V7** (`candidate` : FK `user_id ON DELETE CASCADE`, CHECK `total_capital`/`open_price` > 0 + `pct ∈ (0,100]`, `fills`/`entries`/`exits` JSONB, index `(user_id, trading_date DESC)`) — la colonne `entries` a d'abord été ajoutée en V8 puis **fusionnée dans V7** (feature non publiée, migration immuable pas encore commitée).
- **Couche math pure `candidates.math.ts`** (récupérée + étendue depuis les calculettes shelvées) — fonctions pures, contrat **null-on-bad-input** : `dollarAtRisk`, `entryLadder` (par palier fixe : prix `open×(1+step)`, `maxShares = $risk / ((stop−step)×open)` **arrondi** comme le tableur, zone de sizing `0 < step < stop`), `executionSummary` (risque courant, **résidu** = budget − risque engagé — **sizing seul** depuis la révision), `entrySummary` (entrées réelles en saisie libre → **position moyenne** pondérée + entrée % / risque $ / risque %, **source du cover**), `coverSummary` (% / $ gain vs position moyenne, TP moyen), + `borrowFeePercent` / `gusPercent` rapatriés. Vérifiée sur les chiffres réels du tableur (`casst`, open 12,04 $, budget 365 $).
- **Frontend `features/candidates/`** — page **upsert + dropdown** : sélecteur des candidats du jour → remplit la fiche → ré-enregistre. Formulaire Signal Forms (numériques via `appNumberMask`, date via datepicker), quatre tableaux dérivés en temps réel (échelle d'entrée, suivi d'exécution éditable, **entrées réelles éditables**, échelle de cover éditable ; échelle d'entrée + suivi d'exécution en `mat-expansion-panel` repliables via `StbExpansionModule`), calculettes GUS / borrow intégrées, ticker rendu en chip `stbChip="ticker"`. **Création d'une stat depuis le candidat** : bouton « Créer une stat » ouvrant l'`add-stat-dialog` pré-rempli (date, ticker, gap % issu du GUS, open) → enregistré via `StatsRepository`. Port `CandidatesRepository` + adaptateur `HttpCandidatesRepository` (`core/api/candidates/`), route `/candidates`, item sidenav, i18n FR / EN.
- **Calculettes shelvées absorbées** — les pages standalone `gus-calculator` / `borrow-fee-calculator` (et le `% push` du tableur) **ne sont pas re-commitées** : seule leur logique de calcul est récupérée dans `candidates.math.ts`, le candidat en est désormais le seul propriétaire. Cadrage complet : [`us/candidats.md`](./us/candidats.md).
- **Tests** — back : `CandidateIntegrationTest` (round-trip JSON des ladders, validation 400, lifecycle par date, scope multi-tenant 404). Front : `candidates.math.spec` (formules + null-contract sur les chiffres du tableur), `candidates-page.spec` (chargement du jour, dérivés réactifs, create-vs-update au save).
- **Révisions (2026-06-21)** — (1) nouveau tableau **Entrées réelles** (legs prix + shares libres) qui devient la **source de la position moyenne** ; le suivi à paliers passe **sizing-only** (KPI position moyenne retiré). (2) Tableaux retouchés : colonne « Risque / action » retirée de l'échelle d'entrée, « Shares max » → « Risque $ » dans le suivi (tri ascendant 5 %→35 %), colonnes du cover réordonnées. (3) Échelle d'entrée + suivi d'exécution en **expansion panels**. (4) Migration **V7/V8 fusionnée** + bouton **Purge** Tilt qui nettoie les migrations compilées périmées avant restart.
- **À noter** — le `% push` (présent dans le tableur, jamais codé) est **hors scope v1** (tranché 2026-06-21) : `morningPush` est saisi/persisté mais aucune calculette ne le consomme ; la formule sera reconfirmée en Phase 2.

### ✅ Livré — Toggle devise USD / CAD sur le solde du compte (2026-06-16)

- **Nouveau module backend `forex/`** (hexagonal, léger) — port `ForexRateClient` + adapter unique `FrankfurterForexClient` sur l'API publique [Frankfurter](https://frankfurter.dev) (taux de référence **BCE**, publiés 1×/jour ouvré, **sans clé ni quota**). `ForexService` (façade read-side, normalisation des codes devise) + `ForexController` (`GET /api/forex/rate?base=USD&quote=CAD`). Cache **Caffeine autonome 6 h** dans l'adapter (pas de `CacheManager` partagé — un 2ᵉ bean non qualifié rendrait le `@Cacheable` de `market/` ambigu). Fail-soft : 4xx/5xx/injoignable/devise absente → `UpstreamUnavailableException` (→ 503), jamais de taux inventé. Bloc `forex.frankfurter.base-url` dans `application.yml`.
- **Frontend `features/account/`** — toggle **USD / CAD** dans le hero, à côté du sélecteur de période. La conversion porte **uniquement sur le solde héros** (+ sa variation) ; mouvements, résumé et graphe **restent en USD** (périmètre volontaire). Multiplication **côté client** (`convert()` × taux) — le backend ne vend que le taux, aucune devise ne se propage dans les DTO compte. Légende « 1 USD = 1,3712 CAD · taux BCE, *date* » sous le solde en mode CAD. **Sans persistance** (reset USD à chaque visite), **symbole `$` dans les deux modes** (le toggle + la légende lèvent l'ambiguïté). Si le taux échoue à charger : toggle CAD **désactivé** (tooltip), on reste en USD, **pas de bannière d'erreur** (feature cosmétique). Port + adapter `core/api/forex/`, binding `providers.ts`, i18n FR / EN.
- **Décisions** — taux **quotidien** (et non temps-réel) suffisant pour un solde de compte → provider **keyless** Frankfurter plutôt que TwelveData/FMP (clé + routing mock/réel inutiles). Conversion **présentationnelle côté front** → **zéro migration DB**, compte toujours mono-devise USD en base. Le port `ForexRateClient` laisse la porte ouverte à un provider payant temps-réel (+ routing) sans toucher les appelants.
- **Tests** — back : `FrankfurterForexClientTest` (mapping `rates[quote]` + `date`, fail-soft 5xx, devise absente → échec, cache same-day = 1 seule requête upstream). Front : `forex.http.spec` (surface HTTP, défaut USD→CAD, paramètres + mapping wire ↔ domaine).

### ✅ Livré — Compte broker (solde, mouvements, intégration journal, graphe) (2026-06-16)

- **Module backend `account/`** (hexagonal) — `AccountMovement` (enum Postgres natif `account_movement_type` : `DEPOSIT` / `WITHDRAWAL` / `TRADE` / `ADJUSTMENT`), **balance dérivée** (`Σ amount`, jamais stockée ; `amount` **signé**). Migration **V6** (`account_movement` : FK `user_id` + FK `trade_entry_id ON DELETE CASCADE` + CHECK `amount <> 0` + CHECK `TRADE ⟺ trade_entry_id` + index `(user_id, value_date DESC)` + unique partiel par trade). `AccountService` : ajout dépôt/retrait, **correction de balance** (saisie de la valeur réelle broker → `ADJUSTMENT` du delta signé), édition/suppression des mouvements manuels, **série cumulée end-of-day** ; validation manuelle façon `lexicon` (400), contrat 404-pas-403, `TRADE` read-only sur les endpoints manuels (400). `AccountController` : `GET /movements` (paginé), `GET /summary`, `GET /balance-series`, `POST /movements`, `POST /corrections`, `PUT` / `DELETE /movements/{id}`.
- **Intégration journal → compte** — `TradeEntryService` publie `TradeChangedEvent` (create / update / import, après `saveAndFlush`) ; `TradeMovementSyncListener` (`@EventListener` **synchrone, même transaction**) délègue à `AccountTradeSyncService` qui **upsert / supprime** le mouvement `TRADE` keyé sur `trade_entry_id` (P&L réalisé ≠ 0 → upsert ; null / 0 → suppression). Suppression d'un trade → **cascade DB**. Atomique : trade + mouvement committent ou rollback ensemble. Dépendance cross-contexte `account → journal` via event (même schéma que `market → config`).
- **Frontend `features/account/`** — page **en première position du sidenav** : hero **solde courant** + **variation sur la période** (sélecteur 1S / 1M / 3M / YTD / Tout), **graphe d'évolution**, **mouvements groupés par date** avec sous-total quotidien, panneau **Résumé** (total déposé / retiré, net injecté, P&L trades, corrections, nb). Dialogs ajout / correction (Signal Forms, montant via `appNumberMask`, date via datepicker). `TRADE` read-only avec ticker en chip `stbChip="ticker"` + lien journal. Repository + adapter `core/api/account/`, route `/account`, i18n FR / EN.
- **`StbAreaChart` (`libs/ui`)** — composant SVG autonome (aire + courbe + tooltip au survol), zoneless (signals pures), zéro dépendance externe ; fenêtrage par période + variation calculés **client-side** sur la série `/balance-series` (endpoint pur, date-independent).
- **Décisions** — ledger auditable, **mono-compte v1**, **USD** ; trades poussés **automatiquement** (1 mouvement / trade, **clôturés uniquement**) ; édition manuelle sur `DEPOSIT` / `WITHDRAWAL`. Cadrage complet : [`us/compte-broker.md`](./us/compte-broker.md).
- **Tests** — back : `AccountIntegrationTest` (signe / balance, correction + garde no-op, validation, contrat `TRADE` read-only, scope multi-tenant, série cumulée), `AccountTradeSyncIntegrationTest` (cycle de vie sync journal → compte + cascade DB). Front : `account.http.spec` (surface HTTP + mapping wire ↔ domaine).

### ✅ Livré — stats à parité journal : filtres + recherche + CRUD par propriétaire (2026-06-11)

- **Page `/stats` éditable** — mise à parité avec le journal : barre de **recherche ticker** (debounce 250 ms), **drawer de filtres** (date range + source + plage de gap min/max), et **CRUD** (ajout / édition / suppression). Le CRUD est **réservé aux lignes du propriétaire** (`RADAR` + `MANUAL`) ; les lignes `IMPORT` (communauté) sont en lecture seule — la colonne actions n'affiche les boutons que si `source !== IMPORT`, et le serveur enforce l'ownership (toute ligne non possédée → 404).
- **3e origine `MANUAL`** — `StatSource` passe à `IMPORT` / `RADAR` / `MANUAL`. Le bouton « Ajouter » ouvre un **dialog Signal Forms** (mirror de `add-trade-dialog`) couvrant tous les champs stat (setup + outcome EOD) ; les `%push`/`%LOD`/`%EOD` restent dérivés serveur. Le radar continue de créer en `RADAR` (one-click).
- **Unicité par propriétaire + upsert** — migration **V3** : `CHECK source` étendu à `MANUAL`, **dédup défensive**, puis index unique `ux_stat_entry_day_ticker_owner` sur `(trade_date, ticker, COALESCE(created_by, zero-uuid))`. Une analyse par (jour, ticker, **propriétaire**) : ta ligne (radar/manuelle) et la ligne `IMPORT` communauté du même jour/ticker **coexistent** (comparaison KPI perso vs communauté à venir). Créer pour un (jour, ticker) déjà possédé **écrase** (upsert) ; le ré-import CSV upsert aussi la ligne communauté.
- **Backend** — listing via `JpaSpecificationExecutor` + `StatEntrySpecifications` (visibilité + filtres ticker/date/source/gap). `StatEntryFormRequest` unifié (radar + manuel). Service `create`/`update`/`delete` ownership-scopés (repo `findByIdAndCreatedBy` / `deleteByIdAndCreatedBy`) + upsert (`findByTradeDateAndTickerAndCreatedBy`). Controller : `GET` (filtres) / `POST` / `PUT /{id}` / `DELETE /{id}`. `GlobalExceptionHandler` mappe `DataIntegrityViolationException` → **409** (collision d'édition date/ticker). Sécurité inchangée (seul `/import` reste ADMIN ; CRUD = authenticated, ownership en service).
- **Tests** — back : filtres, upsert, **coexistence user/communauté même jour/ticker**, ownership (édition/suppression d'une ligne `IMPORT` → 404), ré-import upsert. Front : `stats-page.spec` (gating `isOwned`, pipeline delete, flux create + toast 409), `stats.http.spec` (params de filtre + verbes CRUD + `createFromRadar` avec `source`).

### ✅ Livré — radar fiabilisé (price + gap) + « Add stat » par utilisateur (2026-06-11)

- **Float / premarket retirés du radar** — le float venait de FMP `shares-float`, **statique et en retard sur la dilution réelle** sur les small-caps ciblées (ex. GELS : 3.6M chez FMP vs 5.62M sur DilutionTracker). Décision : ne plus afficher de donnée non fiable. Colonnes `float` / `premarketVolume` supprimées de la table, **enrichissement `enrichIfCandidate` retiré** de `MarketScreenerService` (les movers sont persistés bruts ; `TickerFundamentalsClient` & co. restent dormants pour phase 2). Le filtre `applyGusChecklist` ne porte plus que sur **prix $1–$10 + gap ≥ +50 %** (`GUS_CRITERIA` : `floatMin`/`floatMax` retirés).
- **Lien DilutionTracker par ticker** — nouvelle colonne avec lien `target="_blank"` vers `dilutiontracker.com/app/search/<TICKER>` : l'analyse float/dilution reste **humaine**, sur l'outil de référence.
- **Bouton « Add stat » (par utilisateur)** — chaque ligne du radar peut **créer une stat** (modal de confirmation → `POST /api/stats`). Comme une ligne `stat_entry` exige bien plus que `ticker/gap/price` (float, institutions, high/LOD/EOD = données de fin de journée), **le schéma est assoupli** : migration **V2** (`V2__stats_radar_source.sql`) qui passe en **nullable** les colonnes setup + outcome (`float_shares_millions`, `institutions_percent`, les 4 booléens, `high/lod/eod_price`, `push/lod/eod_percent`) ; restent NOT NULL `trade_date`, `ticker`, `gap_up_percent`, `open_price`. Une ligne radar = ligne partielle propre (NULL, pas des 0).
- **Dataset stats : de « global » à « admin-global + privé par user »** — V2 ajoute `source` (`IMPORT` / `RADAR`) + `created_by` (UUID, NULL = jeu admin/global). **Cloisonnement** : `StatEntryRepository.findVisible(userId)` = `created_by IS NULL OR created_by = :me` → un user voit **ses lignes + les lignes admin (import CSV)**, jamais celles d'un autre user. L'**import CSV reste ADMIN-only** (`SecurityConfig` : gate `hasRole("ADMIN")` resserré sur le seul `POST /api/stats/import`), la **création radar est ouverte à tout user authentifié** (`POST /api/stats` retombe sur `/api/**` authenticated). `createFromRadar` pose `source=RADAR`, `created_by=me`, `tradeDate=today` (ET). Export CSV scopé à `created_by IS NULL` → **roundtrip inchangé** (decoder/encoder non touchés, les lignes RADAR partielles ne transitent pas par le CSV).
- **Frontend** — `StatEntry` (modèle + wire) : champs relaxés en nullable + `source` / `createdBy` ; `StatsRepository.createFromRadar` (`POST /api/stats`). Radar : composant `confirm-add-stat-dialog`, flux confirm → create → snackbar succès/erreur, bouton désactivé pendant l'appel. Page `/stats` : nouvelle colonne **Source** (chip, RADAR accentué) + rendu `—` sur tous les champs désormais nullables. **Ticker rendu via la chip directive `stbChip="ticker"`** (radar + stats) — convention posée dans `CLAUDE.md` : tout ticker affiché passe par cette directive.
- **Tests** — back : `StatsListingIntegrationTest` (cloisonnement user/admin, `createFromRadar` partiel `source=RADAR`, défaut date = aujourd'hui), `StatsImportIntegrationTest` (import `source=IMPORT`/`createdBy=null`), `MarketScreenerServiceTest` (movers persistés bruts, plus d'enrichissement). Front : `radar.spec` (filtre price+gap, lien dilution, flux Add stat confirm/cancel/erreur), `stats.http.spec` (mapping nullable + `source`, `createFromRadar`).
