| ⏳ **Pipeline d'analyse — modèle DAG unifié (job, parent/child, cache-aware leaves)** | Ticket fondateur de l'architecture cible décrite dans `docs/metier/vision.md > Le pipeline d'analyse` et `docs/technique/architecture.md > Modèle pipeline d'analyse`. Aujourd'hui le projet a deux tables async séparées (`ticker_narrative_job` Phase 1 + `analysis_job` legacy Phase 0 frozen) sans relation, sans cache visible, sans parent/enfant. La vision cible : **toute analyse devient un nœud d'un DAG**, les feuilles sont les `TickerAnalysis(symbol, day)` et les parents sont les compositions (`PortfolioAggregation`, `WatchlistDigest`, futures). **Schéma cible** : table `job` unifiée — colonnes `id` UUID, `kind` VARCHAR (`TICKER_ANALYSIS` / `PORTFOLIO_AGGREGATION` / `MARKET_REFRESH` / extensible), `parent_id` UUID nullable (FK self pour le DAG), `status` VARCHAR (`PENDING` / `RUNNING` / `DONE` / `DONE_CACHED` / `ERROR` / `CANCELLED`), `origin` VARCHAR (`dashboard` / `cron` / `api` / `parent`), `cache_key` VARCHAR (clé déterministe type `TickerAnalysis:NVDA:2026-05-07`), `target_id` UUID nullable (FK vers `ticker_narrative_snapshot` ou future `portfolio_analysis_snapshot`), `payload` JSONB (input du job — symbol+date pour leaf, portfolio_id+date pour parent), `result_summary` TEXT (« cached snapshot from 09:32 » / « LLM call 8.4 s, retry 0 »), `error` TEXT, timestamps `created_at` / `started_at` / `ended_at`. **Migration** : V6 ou suivante, peut soit (a) créer `job` ex nihilo et migrer les données Phase 1 lignes-par-lignes (clean break), soit (b) garder les tables existantes et créer `job` comme view union pendant une période transitoire. (a) plus propre, (b) moins risqué. **Cache-aware leaves** : chaque feuille `TickerAnalysis(symbol, day)` lookup d'abord `ticker_narrative_snapshot.findFreshFor(symbol, day)` — si hit, status passe direct à `DONE_CACHED` sans appel LLM. Le parent reste `PENDING` tant que tous ses enfants ne sont pas dans un état terminal — implémenté via un listener qui réveille les parents quand un enfant transit. **Tests** : un cas pin par état (cache hit instant, cache miss → RUNNING → DONE, error → retry depuis l'UI, dedup join existing run, parent waits all children terminal, parent runs even if some children ERROR). **Pourquoi pas Temporal / Airflow** : single-user, low-throughput, JVM unique — un thread pool Spring + une table `job` font 95 % du job à 5 % de la complexité. Si SaaS multi-user un jour, Temporal devient pertinent. **Effort estimé** : ~3 jours pour le modèle + l'orchestrateur + le rebranchement Phase 1 ticker narrative dessus + tests. **Prérequis** pour tous les tickets dépendants : « Page Jobs » Phase 3 (UI au-dessus du DAG), « Réintégration Phase 0 » Phase 4 (premier user du DAG côté parent), et indirectement « Décommissionner Phase 0 » Phase 2.5 (qui peut maintenant supprimer les tables legacy sans regret car le DAG les remplace). **Décision recommandée** : ce ticket est un prérequis de design + implémentation à faire **avant** d'attaquer les deux Phase 4 / Phase 3 dépendants |

## Phase 5 — Déploiement

> Aujourd'hui l'app vit uniquement en local via `tilt up` sur le poste de l'utilisateur — aucun accès depuis un autre appareil, aucun moyen d'autoriser quelqu'un d'autre à l'utiliser.
