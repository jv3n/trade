À ré-arbitrer le jour où on a plus de modules ou si on veut booster le côté « outil pro » du projet | 🟢 Basse |

Le user reste sur le dossier, lit la news en ligne, et peut comparer avec le narratif IA juste en dessous. **Approches techniques à explorer** (à arbitrer au moment de l'attaque) : (1) **scraping serveur via Mozilla Readability / jsoup** — fetch la `url`, extrait le contenu principal côté backend, cache court. Limites : sites paywall (Bloomberg, FT, WSJ), sites JS-heavy qui ne rendent rien sans browser headless, ToS souvent restrictifs. (2) **LLM-as-summarizer** — passe la `url` ou le `summary` Finnhub à Claude pour générer un digest 3-5 phrases côté nous. Limites : hallucination quand le LLM "devine" ce que dit l'article sans avoir vu le texte ; coût (~+1 call Claude par item × 10 items = 10 calls / consultation). (3) **Agent récupération + résumé** — pipeline plus sophistiqué (Perplexity-like) : fetch + extract + summarize en une étape contrôlée, optionnellement avec citations vers le source. Plus cher mais qualité supérieure. (4) **Hybride** : pour les sites paywall-free → scraping ; pour les paywalls → LLM summary de title + Finnhub summary (qui est déjà court mais factuel). **Risques transverses** à clarifier avant tout code : (a) **ToS / robots.txt** — scraper Reuters / Bloomberg n'est pas anodin, certains ont des CGU explicites contre l'agrégation. (b) **Coût LLM** — si on utilise Claude pour summariser, prévoir un cache long (24 h+) et un opt-in / opt-out par item. (c) **Qualité variable** — l'effet « narratif IA » risque de se reproduire ici (lecture creuse, halluciné), à observer avant de généraliser. **Dépendances Phase 3** : ce travail recoupe la philosophie observabilité narrative (sait-on de quel article le LLM s'est inspiré ?) et le prompt management. À traiter une fois Phase 3 lancée pour profiter de l'infra de scoring et A/B prompts | 🟢 Basse |

## Dette technique

Sujets identifiés en cours de session, pas bloquants pour la Phase 1 mais à traiter quand l'occasion se présente.

| Sujet | Description | Priorité |
|-------|-------------|----------|
| ⏳ Cleanup des jobs orphelins au démarrage | À chaque hot-reload Tilt (ou crash backend), un job `PENDING` reste `PENDING` à jamais en BDD. `ApplicationReadyEvent` listener qui passe tous les `PENDING` en `ERROR` au boot. ~15 min | 🟡 Moyenne |
| ⏳ Cache Vitest en CI (à mesurer avant) | Aujourd'hui la suite frontend tourne en ~30-60 s en CI dont une fraction de Vitest pur — pas mesuré précisément. Le cache Vitest (transform TS→JS) vit dans `node_modules/.vite/` par défaut, donc wipé à chaque `npm ci`. Pour le rendre cacheable il faut (1) déplacer le cache via `cacheDir: '.vitest-cache'` dans `vitest.config.ts` (ou via `angular.json` si on passe par `ng test`), (2) ajouter une 3e step `actions/cache@v5` sur ce dossier dans `frontend.yml`, key bicéphale `package-lock.json + vitest.config`. **Trigger d'arbitrage** : à attaquer si `npm test` en CI dépasse les 30 s sur la partie Vitest seule (mesurer via les logs du job, pas le total). Tant qu'on est en dessous, le bruit de maintenance (clé de cache à invalider, dépendance sur la config Vitest) coûte plus que le gain. **Préreq avant code** : ajouter une step `time` autour du `npm test` dans `frontend.yml` pour publier la durée Vitest brute sur 5-10 runs avant de décider | 🟢 Basse |
| ⏳ Linter ESLint côté frontend | Aujourd'hui le front n'a que **Prettier** (formatage) — aucune analyse statique. `ng add @angular-eslint/schematics` pour poser la flat config (`eslint.config.js`) + `eslint-config-prettier` pour désactiver les règles formatage qui entrent en conflit avec Prettier. **Ruleset proposé** : `recommended` Angular + TS d'abord, pas de `recommended-type-checked` au démarrage (plus strict mais 5-10× plus lent car nécessite le typage complet) — passage type-checked en session dédiée plus tard quand on veut serrer. **Pipeline** : `npm run lint` ajouté à `package.json`, step CI dans `.github/workflows/frontend.yml` (après le build, avant les tests). Pre-commit hook optionnel via husky + lint-staged — ou plus simple, on laisse la CI bloquer. **Coût caché** : la première passe va probablement remonter ~30-50 warnings sur le code existant (signal vs Observable patterns, missing types, unused vars). Traiter en deux commits : (1) land la config tolérante aux warnings existants, (2) commit dédié `chore(lint): fix initial eslint pass` pour zéroter. ~1-2h en session dédiée | 🟡 Moyenne |
| ⏳ Agent Claude spécialiste doc avec ses propres skills | Configurer un sous-agent Claude Code dédié à la maintenance documentaire. **Cible** : un agent `doc-maintainer` lancé via `Agent` tool quand on touche du code structurant, qui ouvre les bons fichiers (`architecture.md`, `developpement.md`, `ddd.md`, `etat-actuel.md`, `backlog.md`, `commit-conventions.md`, `providers.md`, `ops.md`, `fonctionnalites.md`, `sources.md`) et applique les règles déjà définies dans `CLAUDE.md` section "Documentation" (tableau "fichier ↔ quand le mettre à jour"). **Skills propres** : (1) `doc-cross-check` — détecte les dérives factuelles (modules listés vs réalité du `backend/src/main/kotlin/com/portfolioai/`, repositories listés vs `frontend/src/app/core/`, providers listés vs `application.yml`, workflows listés vs `.github/workflows/`).
